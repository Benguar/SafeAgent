

This project is a  multi layered defense middleware system for AI cognitive firewall, designed to enhance the security and robustness of AI systems against various threats and attacks. The repository includes the following components:

## Tech Stack⚙️:

### Python, FastAPI, scikit-learn, Yaml, Redis, Docker, Rego, LangGraph, Hugging Face, PostgreSQL, Stremlit.


1. **Prompt Scanning**: This endpoint is responsible for analyzing and filtering incoming prompts to identify potential threats or malicious instructions. We use Yaml Based Rules to scan for prompt injection attacks, which can be used to manipulate the behavior of AI systems. The prompt scanning module helps to ensure that only safe and legitimate prompts are processed by the AI system. After scanning of the prompt, we use a scikit-learn trained model for heuristic analysis to further evaluate the safety of the prompt, this acts as a safety net to catch prompt injection attacks  that is not in the policy. This combination of rule-based scanning and machine learning allows for a more comprehensive defense against prompt injection attacks. If safe, we then check for  contents that can be redacted  in the prompt such as PII leaks and to prevent DLP and if found, we sanitize the prompt to remove any potentially content that could contain sensitive information before adding to LLM memory and logging it 

2. **RAG/tool output scanning**: This endpoint is responsible for analyzing the outputs generated by the Retrieval-Augmented Generation (RAG) or tools used by the AI system. Similar to prompt scanning, we use Yaml Based Rules to scan for potential threats or malicious content in the outputs. This helps to ensure that the outputs generated by the AI system do not contain harmful or inappropriate content. If any issues are detected, appropriate actions can be taken, such as blocking the output or flagging it for review. We then use heuristic analysis to further evaluate the safety of the output, this acts as a safety net to catch any malicious content that is not in the policy. After this, we scan to prevent PII and secret leaks in the output and sanitize it,tag it  before sending to LLM memory and then logging it. We ensure that only safe data and information is stored in the LLM memory, set a time to live in redis and we also make sure we limit the number of characters sent to the LLM memory to prevent any potential overflow and make it easier for ReGo to scan and analyze data

3. **LLM output scanning**: This is the final endpoint it is the last layers of defense that no sensitive contents or secrets are sent to the user. This endpoint scans the output generated by the Language model before it is sent to the  user. If it contains any secret or sensitive information, it is redacted and sanitized before being sent to the user and then logged

4. **Dashboard**: The dashboard provides a user-friendly interface for monitoring and managing the security and defense mechanisms of the AI system. It allows users of the SDK to view logs, analyze threats, and configure rules and policies for prompt scanning, RAG/tool output scanning, and LLM output scanning. The dashboard also provides insights into the performance of the defense system and helps users to identify potential vulnerabilities and areas for improvement, the dashboard will be built using streamlit and will be connected to the database to fetch logs and display them in a user-friendly manner.

The SDK will be designed to be centralized,modular and composable and can be easily integrated with various AI systems, providing a unified interface for security and defense mechanisms. It will also be designed to be flexible and customizable, allowing users to configure rules, policies, and thresholds according to their specific needs and requirements.

The SDK will be designed to be lightweight, efficient, performant and scalable to handle large volumes of prompts and outputs without causing significant latency or performance issues.

To reduce latency and improve performance the SDK will:
- Leverage on FastAPI asynchronous capabilities to handle multiple requests concurrently and efficiently.
- Use Redis for LLM memory to save temporary data and information, which allows for fast access and retrieval without the need for frequent database queries.
- Use postgreSQL for logging asynchronously and as a background task in fastapi to avoid blocking the main thread and causing latency issues.
- use redis asynchronously as the LLM memory with TTL to preserve RAM and use redis concurrently to handle multiple requests and data storage efficiently.
- Use Python for prompt normalization, stripping, and leetspeak instead of ReGo because these operations involve string manipulation and require mutability, which is not possible in ReGo due to its immutable string nature. Using Python allows for more efficient and flexible handling of these operations without having to rewrite strings in memory which can spike latency
- Use ReGo for policy engine instead of traditional YAML+Regex approach to scan and analyze data, ReGo is more efficient and faster in processing and analyzing data.It analyses in constant complexity instead of python linear complexity, which helps to reduce latency and improve performance.
- For prompt sanitization and redaction, we will use python and regex not ReGo because strings in ReGo are strictly immutable, and we need to perform operations that require mutability for redaction and sanitization. so even though rego loads rules at constant complexity, Rego will have to rewrite data and rules every time we want to redact or sanitize a prompt, which will lead to increased latency and performance issues. By using python and regex for redaction and sanitization, we can perform these operations more efficiently without the overhead of rewriting data and rules in ReGo even if we have to load the rules at linear complexity, it will still be more efficient than using ReGo for redaction and sanitization.
- will use asyncio.gather for both the OPA engine and heuristic analysis to run concurrently and reduce overall latency, since these operations can be performed independently of each other, running them concurrently allows us to make better use of system resources and reduce the total time taken for both operations.

## Prompt Endpoint performance metrics for 50 word prompt:
- Fastapi request handling time: 0.5ms- 1ms
- python leetspeak stripping, prompt normalization: 0.2ms
- request to OPA port: 1ms
- OPA rego evaluation time: 1.5ms- 3ms    
- ML heuristic analysis: 0.5ms- 2ms
- PII and secret scanning using Regex+ Python: 0.2ms - 0.5ms
- Redis I/O: 0.5ms- 0.8ms

**Total latency**: 3.7ms - 7.3ms depending on the complexity of the prompt and the number of rules evaluated in ReGo, the amount of data to be scanned for PII and secrets and the time taken for heuristic analysis. but since i am using asyncio.gather to run OPA and ML model concurrently, overall latency will reduce by 0.5 - 1ms

**Throughput**: 312-1348 requests per second depends on the amount of workers
(this is using the average for best and worst case latency, and is for 1-4 workers, with more workers the throughput will increase)
